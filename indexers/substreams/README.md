# OnSocial Substreams Indexer

High-performance [Substreams](https://substreams.streamingfast.io/) indexer that
decodes NEP-297 JSON event logs from the OnSocial NEAR contracts and writes them
to PostgreSQL via `substreams-sink-sql`.

## Indexed Contracts

This package indexes **5 contracts**. Other contracts in the workspace
(`manager-proxy-onsocial`, `mock-ft`, `intents-onsocial`, `vesting-onsocial`,
and `staking-onsocial`) are intentionally **not** included here.

| Contract            | Standard       | Event types | Decoder                  | DB sink module   | Schema                     |
| ------------------- | -------------- | ----------- | ------------------------ | ---------------- | -------------------------- |
| `core-onsocial`     | `onsocial` v1  | 5           | [src/core_decoder.rs](src/core_decoder.rs)       | `core_db_out`    | [core_schema.sql](core_schema.sql)         |
| `boost-onsocial`    | `onsocial` v1  | 14          | [src/boost_decoder.rs](src/boost_decoder.rs)     | `boost_db_out`   | [boost_schema.sql](boost_schema.sql)       |
| `rewards-onsocial`  | `onsocial` v1  | 12          | [src/rewards_decoder.rs](src/rewards_decoder.rs) | `rewards_db_out` | [rewards_schema.sql](rewards_schema.sql)   |
| `scarces-onsocial`  | `onsocial` v1  | 7           | [src/scarces_decoder.rs](src/scarces_decoder.rs) | `scarces_db_out` | [scarces_schema.sql](scarces_schema.sql)   |
| `token-onsocial`    | `nep141` v1    | 3           | [src/token_decoder.rs](src/token_decoder.rs)     | `token_db_out`   | [token_schema.sql](token_schema.sql)       |

A combined module (`map_combined_output` / `combined_db_out`) routes all five
contracts through a single stream so a single sink instance can index everything,
keeping concurrent stream usage to one.

## Event Format

All custom contracts emit standard [NEP-297](https://nomicon.io/Standards/EventsFormat)
JSON logs with the prefix `EVENT_JSON:`:

```text
EVENT_JSON:{"standard":"onsocial","version":"1.0.0","event":"DATA_UPDATE","data":[{...}]}
```

The token contract emits the standard NEP-141 events (`ft_mint`, `ft_burn`,
`ft_transfer`).

## Prerequisites

1. Substreams CLI:
   ```bash
   # Linux
   curl -L https://github.com/streamingfast/substreams/releases/latest/download/substreams_linux_x86_64.tar.gz | tar xz
   sudo mv substreams /usr/local/bin/

   # macOS
   brew install streamingfast/tap/substreams
   ```
2. `substreams auth` (free StreamingFast account).
3. Rust wasm target:
   ```bash
   rustup target add wasm32-unknown-unknown
   ```
4. `substreams-sink-sql` for PostgreSQL ingestion.

## Build

```bash
# Compile the WASM module (proto code is generated by build.rs).
cargo build --release --target wasm32-unknown-unknown
```

## Pack `.spkg` Bundles

Each sink variant has its own `.spkg` because the schema and sink module are
embedded in the package. Use [`pack.sh`](pack.sh):

```bash
./pack.sh                # pack all variants (combined + per-contract)
./pack.sh combined       # combined-mode package only
./pack.sh core token     # subset
```

This produces `onsocial-events-vX.Y.Z-{combined,core,boost,rewards,token,scarces}.spkg`.

## Run Locally

Stream a typed module against testnet/mainnet without a sink:

```bash
substreams gui substreams.yaml map_combined_output \
  -e mainnet.near.streamingfast.io:443 \
  --start-block 233084800 \
  -p map_combined_output="core=core.onsocial.near&boost=boost.onsocial.near&rewards=rewards.onsocial.near&token=token.onsocial.near&scarces=scarces.onsocial.near"
```

Per-contract typed modules: `map_core_output`, `map_boost_output`,
`map_rewards_output`, `map_token_output`, `map_scarces_output`.

## Run a SQL Sink

The recommended path is the combined sink launched via [`run_sinks.sh`](run_sinks.sh):

```bash
SUBSTREAMS_ENDPOINT=mainnet.near.streamingfast.io:443 \
DATABASE_URL=postgres://onsocial:onsocial@localhost:5432/onsocial \
NEAR_NETWORK=mainnet \
./run_sinks.sh
```

Set `MODE=per-contract` to run five individual sinks instead (one per contract).

For an in-place upgrade of a database that was populated by an earlier
substreams release, run the SQL files in [`migrations/`](migrations/) once before
reloading Hasura metadata. Fresh deployments only need the current schema files.

## Module Outputs

Per-contract typed maps emit a protobuf `Output` defined in [`proto/`](proto):

- `core_onsocial.v1.Output` — `data_updates`, `storage_updates`, `group_updates`,
  `contract_updates`, `permission_updates`.
- `boost.v1.BoostOutput` — flattened `BoostEvent { payload: oneof ... }`.
- `rewards.v1.RewardsOutput` — flattened `RewardsEvent { payload: oneof ... }`.
- `scarces.v1.ScarcesOutput` — flattened `ScarcesEvent` with operation routing.
- `token.v1.TokenOutput` — NEP-141 `TokenEvent`.

The `*_db_out` modules transform these into
`sf.substreams.sink.database.v1.DatabaseChanges` writes against the matching
`*_schema.sql`.

## Architecture

```
NEAR block
   │
   ▼  EVENT_JSON:{...} log lines (NEP-297 / NEP-141)
src/block_walker.rs ── filter by receiver_id ──► per-contract decoder
                                                │
                                                ▼
                            proto::*::Output (typed events)
                                                │
                                                ▼
                                  *_db_out → DatabaseChanges
                                                │
                                                ▼
                                substreams-sink-sql → PostgreSQL
                                                │
                                                ▼
                                       Hasura GraphQL API
```

## Tests

Rust unit / pipeline tests:

```bash
cargo test
```

SQL schema and view validation against a clean disposable PostgreSQL 16 database:

```bash
make check-substreams-sql
# or, from this directory:
bash scripts/validate_sql.sh
```

This applies the combined deploy schema, migrations, core views, leaderboard
views, and the standalone package schemas without touching a persistent local
database.

End-to-end Hasura/PostgreSQL verification scripts (live testnet):

```bash
HASURA_URL=... HASURA_ADMIN_SECRET=... ./tests/run_all.sh
```
